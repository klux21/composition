Independent RFC-Style Format Specification Of Data Composition 1.0 Format

A Structured, Untyped, Hierarchic Data Composition Format

Version 1.0

Data Composition is a modernized and improved configuration file format
for storing all kind of serialized text data in a form that is easy for
humans and programs to read and write.

1.   Abstract

Data Composition is a minimal, untyped, whitespace-tolerant,
block-structured text format designed for configuration files,
lightweight data serialization, small text databases and lightweight
text base protocols.
The format defines a small set of syntactic constructs: entries,
strings, blocks, sections and comments and intentionally avoids all type
semantics.
The Data Composition format is designed for very small parsers.
Composition does not define executable constructs or encrypted file
formats. Applications interpret string values according their own type
system. A zero-copy parsing is possible.

2.   Conventions Used in This Document

The key words MUST, MUST NOT, SHOULD, SHOULD NOT, and MAY are to be
interpreted as described in RFC 2119.

All examples are non-normative unless explicitly stated otherwise.

3.   Overview

Data Composition documents consists of

  -  zero or more entries
  -  optional top-level values
  -  optional blocks subdocuments
  -  optional top-level sections
  -  optional comments
  -  arbitrary whitespace

Data Compositions are designed to be

  -  easy to parse
  -  easy to write
  -  robust under human editing
  -  suitable for zero-copy implementations
  -  suitable for embedded systems
  -  suitable for configuration files and small text databases
  -  suitable for text-based protocols

The Data Composition format does not define data types.
All values are strings or blocks of Data Composition subdocuments.
Any interpretation of the strings is the responsibility of the
application.

4.   Data Model

Data Composition documents are a sequence of entries.
Each entry consists of

  -  a name (possibly empty)
  -  an optional argument that is either a value
  -  or a block containing a subdocument

Values are either

  -  unquoted strings
  -  quoted strings
  -  blocks

Blocks contain nested entries.
Data Composition documents consists of 8-bit characters.
Application may restrict themselves to UTF-8.

5.   Whitespace Handling (Normative)

The following characters are whitespace according to this standard

  -  0x9   Character Tabulation (TAB)
  -  0xA   Line Feed (LF)
  -  0xB   Line Tabulation (VT)
  -  0xC   Form Feed (FF)
  -  0xD   Carriage Return (CR)
  -  0x20  Space (SP)

This design avoids the complexities of Unicode whitespace handling and
ensures that editors which do not display all control characters cannot
accidentally alter the visible syntactic structure of a Data Composition
document.

Whitespace MAY appear between any two lexical tokens, including before
and after:

  -  names
  -  values
  -  the equals sign (=)
  -  block delimiters ({ and })
  -  comments

Whitespace MUST NOT appear inside unquoted strings.
Whitespace inside quoted strings is preserved exactly as written.
Line breaks have no syntactic meaning except when terminating a line
comment.

6.   Syntax (EBNF)
Code

document        = { entry | comment | section } ;
entry           = named_entry | anonymous_block ;
section         = "[" name "]" ;
named_entry     = name [ value ] ;
anonymous_block = block ;
name            = unquoted | quoted ;
value           = string | block ;
string          = "=" ( unquoted | quoted ) ;
block           = [ "=" ] "{" { entry | comment | section } "}" ;

quoted          = ( "\"" { char | escape } "\"" ) | ( "\'" { char | escape } "\'" ) ;
escape          = "\\" ( "\"" | "\\" | "a" | "b" | "d" | "e" | "f" | "n" | "r" | "t" | "v" | 
                  | oct_escape | dec_escape | hex_escape | byte_escape | char_escape ) ;

oct_escape      = oct_digit [ oct_digit ] [ oct_digit ] ;
oct_digit       = "0"..."7" ;

dec_escape      = "d" dec_digit [ dec_digit ] [ dec_digit ] ;
dec_digit       = "0"..."9" ;

hex_escape      = "x" hex_digit [ hex_digit ] ;
byte_escape     = "u" hex_digit [ hex_digit ] [ hex_digit ] [ hex_digit ]
                  [ hex_digit ] [ hex_digit ] [ hex_digit ] [ hex_digit ] ;
hex_digit       = "0"..."9" | "A"..."F" | "a"..."f" ;

char_escape     = any_char_except( "\0" ) ;

unquoted        = { unquoted_char } ;
unquoted_char   = any_char_except( whitespace, "{", "}", "[", "]",
                  "=", "\"", "#", ";" ) ;

comment         = line_comment | block_comment ;
line_comment    = "#" { not_newline } | ";" { not_newline } ;
block_comment   = "#*" { not("*#") } "*#" | ";*" { not("*;") } "*;" ;

Whitespace is allowed between all tokens except inside unquoted strings.
Any sequence of whitespace is treated as a token separator.

7.   Entries

7.1  Named Entries

A named entry associates a name with an optional value.

Valid forms:
Code

   name
   name = value
   name = "value"
   name = foo
   name { ... }
   name = { ... }

The equals sign (=) is mandatory for string values and optional for
blocks. A named entry without a value is permitted and represents a
name with an empty value. Entries MAY span multiple lines.

Note: Empty values are allowed to support patterns where the presence of
      a name alone conveys meaning. Applications MAY interpret such
      entries as boolean flags or symbolic attributes, but Data
      Compositions themself assign no intrinsic semantics.

7.2  Anonymous Entries

Anonymous entries have an empty name.

Examples:
Code

  { ... }
  = "value"

The name of such an entry is an empty string.
Anonymous entries SHOULD NOT be mixed with named entries in the same
block.
Anonymous string entries SHOULD NOT be used.

8.   Strings

8.1  Unquoted Strings

Unquoted strings consist of any characters except:

  -  whitespace
  -  { } [ ] = " '
  -  comment initiators (#, ;)
  -  NULL (0x00)

Unquoted strings end at the first disallowed character not escaped by a
backslash.

8.2  Quoted Strings

Quoted strings begin and end with a single quote (') or double
quotes ("). Between a pair of single quotes or double quotes the other
quote type becomes a literal character and part of the string data.
Escape sequences within quoted strings are permitted and handled the
same as outside of quoted strings.

8.3  Escape Sequences

Escape sequences allow all names and values to be represented without
requiring surrounding quotes and simplify automated serialization.
They are always treated as part of either a name or a value.
The following escape sequences MUST be supported and interpreted as
follows:

  \"      double quote    (0x22)
  \'      single quote    (0x27)
  \\      backslash       (0x5C)
  \a      bell            (BEL, 0x7)
  \b      back space      (BS,  0x8)
  \t      horizontal tab  (HT,  0x9)
  \n      line feed       (LF,  0xA)
  \t      vertictal tab   (VT,  0xB)
  \r      carriage return (CR,  0xD)
  \f      form feed       (FF,  0xC)
  \e      escape          (ESC, 0x1B)
  \O      byte as a octal value O (1-3 octal digits)
  \dD     byte as a decimal value D (1-3 decimal digits)
  \xH     byte as a hexadecimal value H (1-2 hexadecimal digits)
  \uH     byte sequence according the hexadecimal value H
          (1-8 hexadecimal digits)

Leading zeros within a decimal, octal and hexadecimal escapes are
permitted.
Control characters SHOULD be specified as numeric escapes.
Numeric values MUST NOT exceed byte range nor the number of digits.
Byte sequences SHALL be treated as a common part of a string.
Conforming parsers MUST leave \u interpretation to the application.

8.3.1 Handling of Unknown Escape Sequences (Normative)

Unknown escape sequences MUST be interpreted as
discarding the backslash and keep the following character.

Examples:

  \q      becomes: q
  \*      becomes: *
  \_      becomes: _
  \#      becomes: #
  \;      becomes: ;

A backslash followed by whitespace (e.g. "\ ") becomes that
whitespace but it is strongly discouraged to escape whitespace
but to use numeric or other escapes.

The following exception applies:

A backslash followed by a NULL byte (0x00) is NOT permitted and
terminates the parsing.

9.   Binary Data

Binary data MAY be represented using escape sequences.

Examples:
Code

   "A\x20B"
   "A\xFF"
   "A\x2"
   "A\255"

10.  Blocks of subdocuments

Blocks begin with { and end with }.

Blocks MAY contain:

   - entries
   - comments
   - whitespace
   - sections

Blocks MAY be nested arbitrarily.

A } closes the innermost open block only.
The content of a block is equivalent to a separate Data Composition
document.

11.  Sections

Sections headers indicate the begin of a sequences of section related
entries. A section name has no semantic meaning beyond grouping of
entries. The usage of sections is optional. 
A sections begins with the section header that contains the name
of the section embraced by a pair of square braces [ ].
Leading and trailing comments and whitespaces aren't part of the name.
Whitespaces and comments in the middle of a name are a part of the
section name. The braces of section headers are not nesting. 
 
The sections content begins immediately after the closing brace of
the section header.
A new section implicitly terminates the content of a previous one
of the same documents but not any section in the parent documents.
The end of a document terminates the last top-level section of that
document.
Documents and blocks MAY contain comments and entries before the first
section.
Multiple sections of a document MAY share the same name.

12.  Comments

Comments outside quoted strings are treated as whitespace.
Comments inside quoted strings are literal text.

A NULL byte inside a comment terminates the document.

12.1 Line Comments

Line comments begin with # or ; and extend to the end of the line.

12.2 Block Comments

Block comments begin with #* and end with *# or begin with ;* and end
with *; .
Block comments are not nested.
A block comment that begins with #* or ;* ends with the matching
terminator *# or *;. Block comments MAY span multiple lines.

13.  Parsing Rules

13.1 Generell Parsing Rules

A conforming parser SHOULD
   - provide a function for iterating sections
   - iterate entries until a new section or the document end
   - return block contents as raw text without unescaping
   - treat blocks like independent documents
   - treat whitespaces and comments as token separators
   - treat commas, periods, and escapes as normal token characters
   - treat quoted strings as non-terminating parts of tokens
   - treat escaped characters as literal characters
   - treat a NULL (0x00) as document end at any occurrence
   - provide functions for removing quotes from strings and
     optionally replacing escape sequences

14.   Zero-Copy Model

A conforming parser MAY support zero-copy operation:

   - Names and strings MAY be returned as slices of the input buffer. 
   - The data of blocks MAY be returned as slices of the input
     including the entangling braces for indicating a block content.
   - After an in-place removal of quotes or in-place unescaping the
     modified part of the input buffer shouldn't be parsed again.
   - Applications MAY request an unquoted and optionally unescaped
     copy of names and string values.
   - The content of blocks SHOULD be returned by keeping all quotes
     and escape sequences of the content unchanged.

15.3  Error Conditions

A parser MUST always terminate parsing on:

   - unexpected curled or square braces outside of quoted strings and
     comments
   - any NULL (0x00) bytes in input

16.   Security Considerations

Implementations MUST:

   - avoid names of sections and entries that contain quotes or escapes
   - avoid unbounded recursion in deeply nested blocks if names or
     values refere to blocks or sections
   - avoid memory allocations during unescaping or removing of quotes
   - treat input as untrusted and possibly incorrect
   - ensure that numeric values are treated as decimal numbers except
     they are prefixed by a character sequence that distinguishes a
     number them from decimal numbers by a letter instead of the decimal
     digit '0'. Common prefixes of other bases are 0b for binary,
     0o for octal and 0x for hexa_decimal numbers.

17.   Acknowledgements

The Data Composition format draws inspiration from INI, XML, JSON and
block-structured languages while intentionally avoiding complexity and
type semantics.
